# Harness 反模式与修复方案

在 OpenAI、LangChain 和多个工程团队实践中观察到的常见 Harness 失败模式，附具体修复方案。

## 反模式 1："Markdown 规则"

**症状**：AGENTS.md 写着"不要在 service 层导入 UI 组件"，但 Agent 照做不误。

**根因**：文字说明只是概率性建议。Agent 可能 80% 的时间遵守——意味着 20% 的提交引入违规，随时间累积。

**修复**：将每条 Markdown 规则转化为机械化检查：

| Markdown 规则 | 机械化等价物 |
|--------------|------------|
| "不要从 Y 导入 X" | Dependency-cruiser 规则，构建失败 |
| "所有 API 使用 ResponseWrapper" | ESLint 规则检查返回类型 |
| "新函数要写测试" | Pre-commit hook 中的覆盖率门禁 |
| "遵循命名约定" | 文件/函数名自定义 lint 规则 |

**原则**：如果一条规则重要到值得写下来，那就重要到值得自动化。

## 反模式 2："百科全书式 AGENTS.md"

**症状**：AGENTS.md 有 800+ 行，涵盖项目的方方面面。Agent 要么忽略大部分内容（上下文窗口压力），要么被矛盾/过时的说明搞晕。

**根因**：把 AGENTS.md 当文档写，而非导航索引。

**修复**：重构为 100 行目录：

```markdown
# AGENTS.md（保持在 100 行以内）

## 快速参考
- 技术栈: [你的技术栈]
- 包管理器: [你的包管理器]
- 测试: `[测试命令]` | Lint: `[lint 命令]` | 构建: `[构建命令]`

## 架构 → docs/architecture.md
## API 规范 → docs/api-conventions.md
## 测试策略 → docs/testing.md
## 部署 → docs/deployment.md

## 最常犯的 5 个错误
1. [最常见的错误 + 指向详细文档的链接]
2. ...
```

## 反模式 3："氛围测试"（Vibe Testing）

**症状**：Agent 写代码，人类用肉眼看看，说"看起来不错"。没有自动化测试，没有 CI，没有验证基础设施。

**根因**：速度优先于质量的文化。"AI 很快，坏了重新生成就行。"

**为何危险**：AI 会放大现有模式。第一个没有测试的模块会成为后续所有模块的模板。技术债务以指数级累积，而非线性。

**修复**：最小可行验证：
1. Pre-commit: lint + type-check + 受影响的单元测试
2. CI: 全量测试套件，输出机器可读结果
3. Agent 必须能在本地运行测试并读取结果

---

## 反模式 4："人在回路中的瓶颈"

**症状**：Agent 写代码 → 等人类构建/测试 → 人类反馈结果 → Agent 修复 → 继续等。每轮 30-60 分钟。

**根因**：人类充当 Agent 和构建系统之间的信息中继。

**修复**：
- 通过 WebSocket、CI webhook 或文件监听器将 Agent 直接连接到构建/测试结果
- 人类角色从"信息中继"转变为"验收标准设计者"
- 结果：反馈周期从 30-60 分钟降至秒/分钟级

---

## 反模式 5："非幂等环境"

**症状**：Agent 运行同一个测试两次得到不同结果。Agent 进入无限分析循环，试图调试一个并不存在的代码 bug。

**根因**：环境有副作用——数据库状态变更、文件系统污染、端口冲突、缓存残留。

**修复**：
1. **隔离**：每次测试运行使用新的临时目录
2. **受控变量**：每个测试套件前重置数据库
3. **容器化依赖**：使用 Docker Compose 保证外部服务一致性
4. **确定性种子**：在测试中固定随机种子

**不修复的代价**：某项目经验——12 小时消耗大量 token，很大程度上因为环境非幂等，Agent 持续重复分析同样的非 bug。

---

## 反模式 6："上下文腐烂"

**症状**：Agent 在前几个任务表现良好，但长会话中质量下降。开始遗忘早期决策、自相矛盾或忽略已建立的模式。

**根因**：上下文窗口被执行历史填满，把重要的项目上下文挤出去了。

**修复**：
1. **压缩已完成步骤**：将详细日志替换为单行摘要
2. **外部记忆**：将重要决策写入文件，而非仅存在上下文中
3. **子任务隔离**：每个子任务获得新鲜上下文，仅包含相关信息
4. **定期上下文刷新**：重新注入 AGENTS.md 和当前任务状态

---

## 反模式 7："模式放大"

**症状**：一个糟糕的代码模式（错误的异常处理、不当的抽象、不良的命名）出现在 50+ 个文件中，因为 Agent 把它当模板用了。

**根因**：AI 系统性地复用代码库中发现的模式。坏模式以指数级传播。

**修复**：
1. **尽早捕获**：第一个坏模式实例修复成本低。第 50 个就贵了。
2. **清扫任务**：定期安排 Agent 扫描已知的坏模式
3. **示范代码**：维护 `examples/` 目录，包含 Agent 应遵循的标准好模式
4. **自定义 lint 规则**：机械化地阻止已知坏模式

**来自社区实践**：一个开发者，52 天，97 万行代码。核心教训："第一行坏代码的成本远超你的预期，因为 AI 会把它变成一万行。"

---

## 反模式 8："工具过载"

**症状**：Agent 有 50+ 个工具/MCP 服务器可用，浪费 token 在选择使用哪个上，或选错了工具。

**根因**：一次加载太多工具，没有渐进式披露。

**修复**：
1. **渐进式披露**：按任务阶段加载工具，而非一次性全部加载
2. **Skills 打包**：将相关工具捆绑为逻辑组
3. **工具描述**：为每个工具编写清晰、具体的触发条件
4. **MCP 协议**：标准化工具接口（像 USB-C 一样）以降低认知负担

---

## 反模式 9："缺失的熵管理"

**症状**：项目初期很干净，但逐渐积累不一致——混合命名约定、重复的工具函数、模块间的模式分歧。

**根因**：没有检测和纠正漂移的机制。所有 Harness 投入都用于新代码；没有用于维护现有代码。

**修复**：
1. **定期清扫 Agent**：每周扫描模式漂移、命名违规、死代码
2. **文档同步**：自动检查 AGENTS.md 与实际项目结构是否匹配
3. **一致性 lint**：强制跨模块一致性的规则（而不仅是单文件正确性）
4. **"吐槽墙"**：允许 Agent 标记可疑模式供人工审查，而非默默复制

---

## 快速诊断流程图

```
Agent 产出低质量代码
    │
    ├─ 项目有 AGENTS.md 吗？
    │   └─ 没有 → 反模式 2: 创建一个（使用模板）
    │
    ├─ 有 Agent 能运行的自动化测试吗？
    │   └─ 没有 → 反模式 3: 添加最小可行验证
    │
    ├─ 架构规则是机械化强制执行的吗？
    │   └─ 没有 → 反模式 1: 将 Markdown 规则转化为 lint/CI 检查
    │
    ├─ Agent 能不经人工中转看到测试结果吗？
    │   └─ 不能 → 反模式 4: 将 Agent 连接到 CI/测试输出
    │
    ├─ 测试结果在多次运行间一致吗？
    │   └─ 不一致 → 反模式 5: 修复环境幂等性
    │
    ├─ 长会话中质量下降了吗？
    │   └─ 是 → 反模式 6: 实施上下文管理
    │
    ├─ 坏模式在文件间扩散了吗？
    │   └─ 是 → 反模式 7: 添加清扫任务 + 示范代码
    │
    └─ Agent 搞不清该用哪个工具？
        └─ 是 → 反模式 8: 实施渐进式工具披露
```
