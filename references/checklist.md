# Harness 完备性检查清单

使用此清单审计项目对 AI 辅助开发的就绪程度。每个项目对应一个 CIVC 支柱。评分：已勾选项 / 总项数。

## 评级标准

| 得分 | 评级 | 解读 |
|------|------|------|
| 0-25% | 🔴 危急 | Agent 输出不可靠。技术债务放大风险极高。 |
| 26-50% | 🟡 基础 | Agent 能完成简单任务，但复杂变更会力不从心。 |
| 51-75% | 🟢 良好 | Agent 能有效处理大部分任务，偶尔需要指导。 |
| 76-100% | 🔵 优秀 | Agent 能自主处理复杂任务，可靠性高。 |

---

## 约束（C）——机械化强制执行

- [ ] **C1**: 项目已配置 linter 且通过
- [ ] **C2**: 存在针对项目特定约定的自定义 lint 规则
- [ ] **C3**: Linter 错误信息包含修复建议
- [ ] **C4**: Pre-commit hooks 已安装且强制执行
- [ ] **C5**: 架构依赖方向通过机械化测试
- [ ] **C6**: 类型检查已启用且为严格模式（无 `any` 逃逸口）
- [ ] **C7**: 格式化强制执行已自动化（Prettier/gofmt/Black）

## 告知（I）——结构化上下文

- [ ] **I1**: AGENTS.md（或等价物）存在
- [ ] **I2**: AGENTS.md 在 150 行以内（目录索引，非百科全书）
- [ ] **I3**: AGENTS.md 引用了结构化文档文件
- [ ] **I4**: 架构决策记录在仓库中（非 Slack/Google Docs）
- [ ] **I5**: 常见错误已列出，并附有详细说明的链接
- [ ] **I6**: 构建/测试/部署命令已记录且可直接复制执行
- [ ] **I7**: 目录结构约定已记录

## 验证（V）——自动化校验

- [ ] **V1**: 单元测试存在且可通过单条命令运行
- [ ] **V2**: 测试结果为机器可读格式（JSON/JUnit XML）
- [ ] **V3**: CI 流水线在每个 PR 上运行
- [ ] **V4**: CI 输出对 Agent 可访问（不仅是人类）
- [ ] **V5**: 关键路径存在集成/E2E 测试
- [ ] **V6**: 测试环境是幂等的（相同输入 → 相同输出）
- [ ] **V7**: 代码覆盖率被度量（不要求阈值，但需跟踪）

## 纠正（R）——错误恢复

- [ ] **R1**: 错误信息包含文件路径和行号
- [ ] **R2**: 错误信息尽可能包含修复建议
- [ ] **R3**: 测试失败能区分超时和代码错误
- [ ] **R4**: Agent 修复后能重新运行失败的测试
- [ ] **R5**: 间歇性失败的测试已识别并隔离
- [ ] **R6**: CI 失败触发清晰的结构化通知

## 熵管理（E）——持续健康

- [ ] **E1**: 定期代码质量扫描已安排（手动或自动）
- [ ] **E2**: 文档与代码保持同步（有检查或自动化）
- [ ] **E3**: 死代码检测和清理流程已建立
- [ ] **E4**: 跨模块的模式一致性被强制执行
- [ ] **E5**: 技术债务被跟踪并排定优先级

---

## 优先实施顺序

对于从零开始的项目，按以下顺序实施以获得最大收益：

### 优先级 1（快速见效）
1. **I1+I2**: 创建 AGENTS.md 作为简洁的目录索引
2. **C1+C7**: 配置 linter 和格式化工具
3. **C4**: 安装 pre-commit hooks
4. **I6**: 记录构建/测试/部署命令

### 优先级 2（验证体系）
5. **V1+V2**: 确保测试存在且输出结构化结果
6. **V3**: 搭建 CI 流水线
7. **R1+R2**: 改善错误信息，添加修复提示
8. **V6**: 修复非幂等的测试环境

### 优先级 3（深化）
9. **C2+C3**: 编写带 Agent 友好信息的自定义 lint 规则
10. **C5**: 添加架构依赖测试
11. **V5**: 为关键路径添加 E2E 测试
12. **I3+I4**: 将文档结构化为被引用的文件

### 持续进行
13. **E1-E5**: 建立熵管理实践
14. **V7**: 跟踪覆盖率趋势
15. **R5**: 隔离间歇性失败的测试
