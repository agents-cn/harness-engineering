# CIVC 框架详解

CIVC（约束 Constrain、告知 Inform、验证 Verify、纠正 Correct）框架将 Harness 的所有职责组织为四个互补的支柱。

## 1. 约束（Constrain）——机械化强制执行

### 原则

文字说明（"不要从 Y 导入 X"）只是建议。构建失败才是规则。目标是通过工具而非文档，使无效状态无法被表示。

### 实施模式

#### 自定义 Linter 规则

创建项目特定的 lint 规则，其错误信息直接包含修复建议：

```
ERROR: Service 层导入了 UI 组件（违反依赖方向）
  at src/services/userService.ts:3
  FIX: 将共享类型移至 src/types/ 并从那里导入。
  SEE: docs/architecture.md#dependency-rules
```

各生态系统的工具：
- **TypeScript/JS**: ESLint 自定义规则、`dependency-cruiser`
- **Go**: `go vet` 自定义分析器、`depguard`
- **Python**: `pylint` 自定义检查器、`import-linter`
- **Java/Kotlin**: ArchUnit
- **Rust**: `clippy` 自定义 lint、`cargo-deny`

#### 依赖方向测试

定义并测试允许的依赖流向（适配你的项目分层）：

```
// 示例分层——根据你的项目架构自定义
Types → Config → Repository → Service → Runtime → UI
```

任何反方向的导入都应导致构建失败。

#### 预提交门禁

阻止违反约束的提交：
1. Lint（语法 + 自定义规则）
2. Format（统一风格）
3. Type-check（类型安全）
4. 架构测试（依赖方向）

### 行业实证

**OpenAI**（100 万行零人工代码）：使用确定性 linter 和结构化测试来机械化地强制执行架构。违反依赖规则的 Agent 会立即收到带修复说明的构建失败。

**LangChain**：同一个模型（Claude），不同的 Harness——在 Terminal Bench 2.0 上成功率从 52.8% 跃升至 66.5%，主要归功于更好的约束执行。

**行业实践：幂等优先原则**——某大规模 Go 后端团队发现，非幂等环境会导致 Agent 进入无限分析循环。约束不仅是"遵循模式"，而是"环境物理上阻止了非幂等操作"。将数据库迁移、文件操作等设计为幂等，是约束层的底层基础。

## 2. 告知（Inform）——结构化上下文交付

### 原则

Agent 只能使用它能发现的东西。仓库内容是唯一的信息源。Slack、Google Docs、走廊对话——对 Agent 来说都不存在。

### AGENTS.md 设计

**反模式**：800 行的巨型文件，面面俱到。

**最佳实践**：100 行的目录索引，指向结构化文件：

```markdown
# AGENTS.md

## 项目概览
简要的 2-3 句描述。

## 架构
See: docs/architecture.md

## API 规范
See: docs/api-conventions.md

## 测试策略
See: docs/testing.md

## 最常犯的 5 个错误
1. 不要在 UI 层导入 services → 见 architecture.md
2. 所有日期用 UTC，仅在 UI 层转换 → 见 api-conventions.md
3. 用项目指定的包管理器 — 检查 lock 文件类型
4. 修改 API schema 后运行代码生成命令
5. E2E 测试需要先启动依赖服务
```

### 渐进式披露

按需加载上下文，而非一次性全部推送：

1. **第一层**：AGENTS.md 目录（约 100 行，始终加载）
2. **第二层**：被引用的文档（在相关领域工作时加载）
3. **第三层**：代码示例和详细规范（在特定需求时加载）

这样可以避免上下文窗口污染，让 Agent 保持专注。

### 上下文腐烂预防

长时间运行的 Agent 会话会出现上下文退化。三个对策：

1. **压缩已完成步骤**：将详细执行日志替换为单行摘要
2. **外部记忆**：将重要状态持久化到文件，而非仅依赖上下文窗口
3. **子任务隔离**：每个子任务获得独立的新鲜上下文，仅包含相关信息

### 行业实践

**MCP 协议标准化**——通过统一工具接口（类似 USB-C 的思路），配合渐进式工具披露和 Skills 打包，避免 Agent 面对过多工具时的决策瘫痪。

**状态持久化实践**——通过文件系统 + Git 实现跨会话的状态持久化；结合 Bash 脚本提供动态编程能力，支持 Agent 自主问题解决。关键洞见：Agent 的"记忆"不应只存在于上下文窗口中。

## 3. 验证（Verify）——自动化校验

### 原则

验证必须产出**物理结果**（通过/失败、exit codes、截图），而非意见。Agent 需要机器可读的信号来进行自我纠正。

### 三层验证架构

| 层级 | 不确定性类型 | 反馈速度 | 实现方式 |
|------|------------|---------|---------|
| **隔离层** | 逻辑错误、API 契约 | 秒级 | 临时目录、受控变量、容器化依赖 |
| **进程层** | 操作系统行为、进程交互 | ~10 秒 | 真实进程交互——避免使用 mock 掩盖系统约束 |
| **栈层** | 网络时序、并发竞争 | 分钟级 | 端到端：浏览器 + 服务 + 依赖全链路 |

### 四层反馈回路

| 层级 | 触发时机 | 速度 | 捕获内容 | 工具示例 |
|------|---------|------|---------|---------|
| **L1 即时反馈** | 代码变更（<1s） | 秒级 | 语法、类型 | 热重载、watch 模式、IDE lint |
| **L2 预提交** | 提交前（~5min） | 分钟级 | 业务逻辑回归 | 单元测试、pre-commit hooks |
| **L3 集成** | 创建 PR | ~10 分钟 | 多模块交互 | E2E 测试、Agent 代码审查 |
| **L4 全量回归** | CI 流水线 | 小时级 | 完整回归 + 多平台 | GitHub Actions / GitLab CI |

**核心原则**：尽可能在 L1 层拦截——频率最高、成本最低。每跳过一层都会创建一条未检查的路径。

### 可观测 Agent

将 Agent 接入运行时可观测性：
- Chrome DevTools（视觉输出验证）
- 日志系统（错误模式检测）
- 指标看板（性能回归）
- 截图比对（UI 漂移检测）

### 行业实践

**自动化反馈回路加速**——某跨端开发团队通过 WebSocket 在 Agent 与模拟设备之间建立实时连接，将单轮反馈从 30-60 分钟降至分钟级（10x+），总开发周期从 2 天降至 4 小时（6x）。关键：让 Agent 能"看到"运行结果。

**结构化任务编排**——某大规模代码迁移项目中，3 天完成 9 万行代码迁移（同一任务之前失败了 3 次）。成功关键：结构化任务编排 + 基于经验的反馈回路，而非让 Agent 自由发挥。

## 4. 纠正（Correct / R）——结构化错误恢复

### 原则

当验证失败时，Agent 需要**可操作的、机器可读的**反馈来进行自我修复。含糊的日志消息浪费 token；结构化的错误响应加速恢复。

### 信号清晰度

```
❌ 差：  "Error: something went wrong in the pipeline"
✅ 好：  {"status": "FAIL", "test": "user-auth-flow", "error": "Expected 200, got 401",
          "file": "src/auth/handler.ts:42", "fix_hint": "Check JWT expiry logic"}
```

纠正信号的要求：
- **机器可读格式**：JSON、exit codes、结构化日志
- **位置明确**：文件路径 + 行号
- **修复建议**：错误信息中内联修复指导
- **分类清晰**：timeout（外部问题） vs. failure（代码 bug） vs. flaky（值得重试）

### Ralph Loop

拦截 Agent 过早退出。当 Agent 说"我已尽力"但测试仍在失败时：

1. 重新注入失败的测试结果
2. 明确声明："任务尚未完成。这些测试仍在失败。"
3. 提供具体的失败上下文
4. 设置最大重试次数以防止无限循环

### Timeout vs. Failure 区分

| 信号 | 含义 | Agent 行为 |
|------|------|-----------|
| Exit code 1 | 代码错误 | 分析错误，修复代码，重试 |
| Exit code 124 | 超时 | 报告为环境问题，不修改代码 |
| Exit code 2 | 缺少依赖 | 安装依赖，重试 |
| Flaky（间歇性） | 非确定性 | 重试一次，然后标记待人工审查 |

### 行业实践

**纠正基础设施的一次性投入**——某项目经验：12 小时消耗大量 token，但这笔开销是在构建可复用的验证基础设施，而非经常性支出。核心洞见：在纠正基础设施上一次性投入，在所有后续 Agent 工作中分摊。

## 跨支柱交互

四大支柱并非独立——它们形成相互增强的反馈循环：

```
约束 → 定义验证检查的内容
告知 → 告诉 Agent 约束的内容和原因
验证 → 产生纠正处理的信号
纠正 → 可能更新告知（新文档）或约束（新规则）
```

成熟的 Harness 四大支柱全部激活且相互连接。部分实现会产生盲区：
- 有约束无验证 = 规则存在但违规未被发现
- 有验证无纠正 = 问题发现但未被修复
- 有告知无约束 = 指南存在但无法执行
- 有纠正无告知 = 修复了问题但未沉淀经验
